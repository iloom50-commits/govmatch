# AI 상담 재설계 — 전체 아키텍처 (00_overview)

**작성일**: 2026-04-21
**목적**: 누적 프롬프트 패치 실패 반복 → 근본 설계 재출발

---

## 1. 배경: 왜 재설계하는가

### 1.1 현재 상태
- AI 품질 점수 평균 **5.2~5.5 / 10** (B 초반)
- 4회 개선 시도에도 **플랫 또는 하락**
  - Level 1+2+3 프로필 필터
  - 5대 품질 규칙 (Perplexity/Cursor 패턴)
  - 신호 감지 4종 (반복/심화/주제전환/중복)
  - Few-shot 예시 4개
- 측정 variance가 개선 효과보다 큼 → 실질 개선 불확실

### 1.2 누적 실패의 구조적 원인

**모든 패치가 "프롬프트에 텍스트 추가" 방식**:
- LLM attention 분산 → 규칙 선택적 따름
- 프롬프트 길어지며 서로 상쇄
- JSON 구조화는 프롬프트로 불가 (LLM 한계)

**대표 버그**: PRO 상담 중 AI가 "2019년 설립" 발화에서 읽었지만 `collected.establishment_date`에 저장 실패 → 매칭 트리거 안 됨 → 추가 질문 반복.

### 1.3 재설계의 핵심 교훈

> **프롬프트는 "제안"이고, 코드·API·DB는 "사실"이다.**

→ LLM에 맡길 일과 코드에 맡길 일을 **명확히 분리**한다.

## 2. 최상위 원칙 (사장님 확정)

### 2.1 근본 목적
> **"매칭 대상의 정보를 명확히 파악하여 정확하게 매칭하는 것"**

### 2.2 운영 원칙
1. **일상 대화 지원 안 함** — 범위 밖이면 "저는 ○○○ 전문 상담자입니다" 응대
2. **정보 수집 방식 분리**:
   - PRO: **폼 기반** (확정적)
   - LITE: **자연어 + 기존 프로필 활용** (유연)
3. **보고서는 수동 버튼** (자동 생성 X)
4. **LITE ↔ PRO 공고상담 분리** (공유 엔진 중단)

### 2.3 설계 원칙 (이번 재설계 한정)

**원칙 1: LLM 역할 최소화, 코드 역할 최대화**
- LLM: 자연어 이해·요약·설명
- 코드: 상태 관리, 데이터 구조화, 분기, 검증

**원칙 2: Schema 우선**
- 모든 AI 응답은 Gemini `response_schema`로 **구조 강제**
- Post-processing(regex/NER)으로 누락 보완

**원칙 3: 프롬프트는 짧고 단일 책임**
- 현재 평균 3000+ 토큰 → 목표 500~1000 토큰
- 규칙보다 Few-shot 예시가 효과 큼

**원칙 4: 상태 기계(FSM) 명시**
- 대화 상태·전이 조건을 **코드로 명시**
- 프롬프트가 아닌 상태 코드가 플로우 주도

## 3. 시스템 경계: LITE vs PRO

### 3.1 본질적 차이

| 구분 | LITE | PRO |
|---|---|---|
| **사용자** | 일반인·기업 본인 | 컨설턴트 (≠ 고객) |
| **목적** | 내게 맞는 것 찾기 | **고객을 위한 보고서 만들기** |
| **본질** | 대화형 탐색 | **보고서 생성 파이프라인** |
| **정보 수집** | 기존 프로필 + 대화 보완 | **폼 입력** |
| **최종 산출물** | 공고 리스트 + 판정 대화 | 보고서 (문서) + 상담 기록 |
| **성공 지표** | 대화 만족도 | **보고서 정확도 + 컨설턴트 효율성** |

### 3.2 왜 이 구분이 중요한가

지금까지 PRO를 **"복잡한 LITE"**로 취급했음 → 실패.
PRO는 근본적으로 **문서 생성 도구**이지 대화 도구가 아님.

## 4. 6개 AI 에이전트 구성

### 4.1 LITE 영역 (3개)

#### ① LITE 공고상담 AI
- **미션**: 유저가 선택한 공고에 대해 "나에게 해당되나?" 판정
- **입력**: 유저 프로필 (DB) + 공고 상세 (`announcement_analysis`)
- **상태**: 단일 세션 (대화)
- **출력**: 자격 판정 + 근거 + 대안 제안
- **상세**: [`01_lite_announce.md`](./01_lite_announce.md)

#### ② LITE 정책자금 AI
- **미션**: 유저의 자금 니즈에 맞는 정책자금·융자·보증 공고 추천
- **입력**: 유저 프로필 + 자연어 요청
- **상태**: 대화 흐름 (수집 → 추천 → 심화)
- **출력**: 맞춤 공고 리스트 + 비교
- **상세**: [`02_lite_fund.md`](./02_lite_fund.md)

#### ③ LITE 보고서 생성
- **미션**: 상담 내용을 문서로 정리 (유저 본인 보관용)
- **트리거**: 유저가 `📄 보고서 만들기` 버튼 클릭
- **포맷**: 1~2페이지 간단 요약
- **상세**: [`03_lite_report.md`](./03_lite_report.md)

### 4.2 PRO 영역 (3개)

#### ④ PRO 매칭 AI (기존 유지, 정보수집만 폼으로)
- **미션**: 고객 프로필 → 매칭 → 연속 상세 상담 → 종합 보고서
- **정보수집**: **폼** (이전: 자연어)
- **상태**: Mode A (매칭) → Mode B (상담)
- **연속성**: 개별 공고 상담이 **하나의 세션**에서 이어짐
- **상세**: [`04_pro_match.md`](./04_pro_match.md)

#### ⑤ PRO 공고상담 AI (신규)
- **미션**: 컨설턴트가 **알고 있는 특정 공고**에 대해 고객 대신 심화 상담
- **LITE 공고상담 대비 차이**:
  - 전문가 세부 내용 추가 (심사 기준 가중치, 실제 선정률, 신청 꿀팁, 유사 사업 비교)
  - 답변 대상이 "고객"인 구조
  - 답변 포맷이 "컨설턴트가 고객에게 설명용"
- **상세**: [`05_pro_announce.md`](./05_pro_announce.md)

#### ⑥ PRO 보고서 생성
- **미션**: 상담 전체를 고객 전달용 전문 보고서로 정리
- **트리거**: 컨설턴트가 `📄 보고서 생성` 버튼 클릭
- **포맷**: 5~10페이지 전문 문서 (PDF)
- **상세**: [`06_pro_report.md`](./06_pro_report.md)

## 5. 공통 컴포넌트

### 5.1 정보 수집·저장 계층
- **기존 프로필 조회** (users 테이블, 52~100% 채움)
- **LITE 보완 인터뷰**: 상담 시작 시 1~2 필드만 자연어로 빠르게 수집 + DB 저장
- **PRO 폼**: 신규 고객은 폼 필수, 기존 고객은 저장된 프로필 불러오기
- **자동 저장**: 대화 중 새로 얻은 정보는 `users`/`client_profiles` 테이블에 즉시 저장
- **상세**: [`07_data_flow.md`](./07_data_flow.md)

### 5.2 보고서 생성 공통 엔진
- LITE 보고서 / PRO 보고서 모두 같은 내부 엔진 사용
- 차이는 **템플릿**만 다름 (`reports/templates/lite_brief.j2`, `reports/templates/pro_full.j2`)
- 데이터 파이프라인 통합

### 5.3 테스트 전략
- 각 AI 에이전트마다 **맞춤 페르소나 3~5명**
- 미션 달성 여부를 **명확히 측정 가능한 지표**로 정의
- Gemini + OpenAI 이중 평가
- 다회 실행 평균 (variance 완화)
- **상세**: [`08_test_strategy.md`](./08_test_strategy.md)

## 6. 데이터 플로우 (고수준)

```
┌─────────────────────────────────────────────────────────┐
│                      사용자 진입                           │
└─────────────────────────────────────────────────────────┘
              │                              │
         [LITE 유저]                   [PRO 컨설턴트]
              │                              │
    ┌─────────┴──────────┐       ┌──────────┴──────────┐
    ▼                    ▼       ▼                     ▼
[공고 탐색]          [자금 질문]  [신규 고객 폼]     [기존 고객 선택]
    │                    │       │                     │
    ▼                    │       ▼                     │
[공고 클릭]              │   [프로필 입력/불러오기] ◀──┘
    │                    │       │
    ▼                    │       ▼
[LITE 공고상담 AI①]   [LITE 정책자금 AI②]   [PRO 매칭 AI④]
    │                    │       │
    │                    │       ▼
    │                    │   [매칭 결과]
    │                    │       │
    │                    │       ▼
    │                    │   [공고 클릭]
    │                    │       │
    │                    │       ▼
    │                    │   [PRO 공고상담 AI⑤]
    │                    │       │
    ▼                    ▼       ▼
┌──────────────┐    ┌──────────────┐
│ LITE 보고서③  │    │ PRO 보고서⑥   │
│ (수동 버튼)   │    │ (수동 버튼)    │
└──────────────┘    └──────────────┘
```

### 핵심 데이터 흐름 원칙
1. **수집된 정보는 즉시 DB 저장** (대화 중 얻은 것 포함)
2. **기존 프로필은 상담 시작 시 자동 주입**
3. **공고 매칭 결과는 세션에 저장** (보고서 생성 재료)
4. **보고서는 별도 테이블에 영구 저장** (재조회 가능)

## 9. 구현 로드맵 (고수준)

### Phase 1: 문서 완성 (2~3시간)
- 01~08 문서 작성
- 사장님 검토·수정

### Phase 2: 기반 계층 (4~6시간)
- Schema 정의 (`ai_schemas.py`)
- FSM 엔진 (`consult_fsm.py`)
- 프롬프트 분리 (`prompts/` → `.md` 또는 구조화 Python)

### Phase 3: AI별 구현 (각 2~4시간)
- 1개 AI씩 순서대로
- 각 AI 구현 후 **즉시 페르소나 테스트**
- 기준 통과 시 다음 AI로

### Phase 4: 통합 테스트 (3~4시간)
- 전체 플로우 E2E 테스트
- 다회 실행 variance 측정
- 기존 대비 점수 비교

### Phase 5: 점진적 전환 (A/B)
- Feature flag로 신규 AI 사용 비율 조절
- 전환 기간 중 기존 AI 병행 유지
- 사용자 피드백·지표 기반 전환 결정

| B6 | 선택지 버튼이 대화 맥락과 무관 | AI choices + 기본 choices 병합 로직 재설계 |

## 11. 이 재설계가 성공할 것이라 믿는 이유

1. **문제 원인을 이번엔 정확히 파악** (프롬프트로 풀 수 없는 구조적 한계)
2. **LLM 제약을 인정**하고 코드가 책임질 부분을 늘림
3. **폼 도입**으로 정보 수집 실패를 구조적으로 해결
4. **Schema 강제**로 JSON 구조화 신뢰성 확보
5. **수동 패치 누적이 아닌 일관된 설계**로 시작

## 12. 실패할 수 있는 리스크

1. **과도한 설계로 인한 구현 지연** → 단계적 접근 (Phase 2~3 분리)
2. **LITE 유저의 폼 저항** → PRO만 폼, LITE는 자연어 유지로 완화
3. **Gemini Schema 복잡도 제한** → 단순 스키마로 시작, 점진 확장
4. **페르소나 테스트 variance** → 다회 실행 평균 + 정량 KPI 추가
