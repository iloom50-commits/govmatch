# 프로젝트 작업 규칙

## 작업 전 확인 필수

사용자가 요청을 하면, **코드 작성이나 실행에 들어가기 전에** 반드시 아래 절차를 따른다:

1. 요청 내용을 내가 이해한 대로 **간결하게 정리**하여 보여준다
2. 작업 범위, 영향받는 파일/기능, 예상 결과를 명시한다
3. 사용자가 **"맞아" / "진행"** 등으로 확인한 후에만 작업을 시작한다

잘못된 이해로 엉뚱한 작업을 하는 것을 방지하기 위한 필수 프로세스이다.

## 지시 의도 재확인 (절대 원칙)

사용자의 모든 지시에 대해 **작업을 시작하기 전에 반드시** 아래를 수행한다:

1. **지시 내용을 간략히 정리**하여 "지금 요청하시는 것은 ~~~ 이 맞나요?" 형식으로 되묻는다
2. **의도(왜 이걸 원하는지)**까지 포함하여 확인한다 — 단순히 명령어를 앵무새처럼 반복하지 않는다
3. 사용자가 확인(`맞아` / `진행` / `응` 등)하기 전에는 **어떠한 코드 수정·실행·커밋도 하지 않는다**
4. 확인 없이 작업을 진행하는 것은 **규칙 위반**이다

예시:
- ❌ "알겠습니다, 수정하겠습니다." → 바로 작업
- ✅ "지금 요청하시는 건 **A 파일의 B 함수에서 C 로직을 D로 바꾸는 것**으로 이해했습니다. 의도는 **E 문제를 해결**하기 위함이 맞나요? 확인 주시면 진행하겠습니다."

이 원칙은 단순/명확해 보이는 요청에도 적용된다 — 작은 오해가 큰 재작업으로 번지는 것을 막기 위함이다.

## 검증 보고의 정직성 (절대 원칙)

테스트·검증 결과를 보고할 때, **실제로 수행한 것만 보고한다.**

1. **하지 않은 검증을 했다고 말하지 않는다** — 코드 경로 확인(정적 분석)과 실제 동작 확인(런타임 테스트)은 다르다. 구분해서 보고한다.
2. **할 수 없는 상황은 솔직히 말한다** — 예: "소셜 로그인 계정이라 API 토큰을 발급할 수 없어 실제 API 호출 테스트는 불가능합니다"
3. **검증 범위를 명시한다** — 무엇을 확인했고, 무엇을 확인하지 못했는지 구분한다.
4. **추측을 사실처럼 포장하지 않는다** — "정상 동작할 것입니다"와 "정상 동작을 확인했습니다"는 완전히 다르다.

예시:
- ❌ "5가지 페르소나로 전체 플로우를 검증했습니다" (실제로는 문자열 조립만 확인)
- ✅ "코드 경로상 client_id가 전달되는 것을 확인했습니다. 실제 AI 응답이 고객 정보를 반영하는지는 API 호출이 불가하여 미확인입니다."
